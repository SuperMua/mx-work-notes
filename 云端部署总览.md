# MX工作便签云端部署总览

这份总览文档用于帮助你快速判断：当前项目应该采用哪一种云端部署方式，应该先看哪一份详细文档，以及每种方案分别适合什么场景。

MX工作便签当前由三部分组成：

- `apps/web`：前台用户端
- `apps/admin`：后台管理端
- `apps/api`：NestJS API 服务

推荐的总体原则保持不变：

- 前台和后台优先作为静态站点发布
- API 作为独立 Node 服务运行
- PostgreSQL 优先使用独立数据库实例或托管数据库
- 正式环境统一接入 HTTPS

## 2. 选型建议

### 2.1 如果你是正式交付客户

优先推荐以下两种：

- 有云厂商偏好时：选择阿里云版或腾讯云版
- 没有云厂商偏好，但希望后期最稳：选择 Ubuntu + Nginx + systemd 实战版

### 2.2 如果你希望部署快、界面化操作多

优先选择：

- 宝塔面板版

这个方案更适合：

- 中小团队
- 演示环境
- 不想手写太多 Linux 运维命令

### 2.3 如果你希望前端部署最轻、更新最快

优先选择：

- 前台静态托管 + API 独立部署版

这个方案更适合：

- 前台和后台交给静态托管平台
- API 单独放在云服务器或托管 Node 平台
- 后续前端更新频率较高

## 3. 推荐的正式架构

不管你最终选哪一种文档，正式上线都建议尽量靠近下面这套结构：

- 前台：独立域名，例如 `app.example.com`
- 后台：独立域名，例如 `admin.example.com`
- API：独立域名，例如 `api.example.com`
- 数据库：托管 PostgreSQL，或至少与应用分离

推荐原因：

- 前台、后台和 API 边界清晰
- 更容易做证书、反向代理和缓存策略
- 后续排障和扩容更简单

## 4. 当前项目的生产注意事项

### 4.1 前端环境变量是构建时写入

前台和后台当前都只保留一个前端环境变量：

```bash
VITE_API_BASE_URL=https://api.example.com
```

这意味着：

- API 域名改了，前台和后台必须重新构建
- 静态站点不能只改服务器配置就自动切换 API 地址

### 4.2 `seed` 在生产环境会跳过

当前 `apps/api/prisma/seed.ts` 在 `NODE_ENV=production` 时会直接跳过执行。

这意味着：

- 正式生产环境不应依赖 `db:seed` 生成默认账号
- 临时验收环境如果确实需要默认账号，可以在初始化阶段临时不用 `NODE_ENV=production` 执行一次 seed
- 执行完成后必须修改默认密码

### 4.3 API 运行方式

当前 API 生产启动入口为：

```bash
node apps/api/dist/apps/api/src/main.js
```

正式环境建议使用：

- `systemd`
- PM2
- 或云平台自己的守护进程能力

不建议长期直接在前台终端裸跑。

## 5. 部署前必须准备的内容

无论哪一种云端方案，建议先准备：

- 可访问的 PostgreSQL 实例
- 一套正式域名或至少 3 个子域名
- HTTPS 证书
- API 运行环境
- 前后台静态产物发布位置

建议提前确定：

- 前台域名
- 后台域名
- API 域名
- 数据库所在地域
- 服务器所在地域

## 6. 上线前统一检查项

所有方案上线前都建议完成下面这些检查：

- `https://api.example.com/health` 可访问
- 前台登录页可访问
- 后台登录页可访问
- 前台可登录
- 后台可登录
- 前台能创建、编辑、归档、删除便签
- 后台能打开数据看板、账号管理、工作区管理、便签管理
- 前后台访问 API 走 HTTPS
- 数据库已开启备份
- 若使用过默认账号，已修改密码
