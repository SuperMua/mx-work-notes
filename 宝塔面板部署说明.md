# MX工作便签宝塔面板部署说明

本文档适用于希望通过宝塔面板完成站点管理、反向代理、证书配置和 API 进程托管的场景。

## 1. 适用场景

宝塔面板方案适合：

- 希望少写 Linux 命令
- 需要界面化管理站点和证书
- 部署演示环境、试运行环境或中小规模正式环境

推荐结构如下：

- 前台：宝塔站点 1
- 后台：宝塔站点 2
- API：Node 进程 + 宝塔反向代理
- 数据库：优先使用独立 PostgreSQL，不建议长期把数据库和应用混在同一台机器

## 2. 资源建议

建议准备：

- 1 台 Ubuntu 云服务器
- 宝塔面板
- Nginx
- Node.js 运行环境
- PostgreSQL 数据库实例
- 3 个子域名

如果是正式环境，建议数据库独立于宝塔服务器。

## 3. 安装宝塔前的准备

建议先完成：

- 云服务器安全组放行 `22`、`80`、`443`
- 服务器解析到域名
- 准备好数据库连接信息

## 4. 安装宝塔和基础软件

安装宝塔后，优先在面板内安装：

- Nginx
- Node.js 管理器或 Node 运行环境
- PM2 或进程守护工具

如果面板里没有 PostgreSQL 管理能力，也没关系，因为当前项目更推荐使用独立云数据库。

## 5. 上传或拉取代码

推荐把项目放在：

```text
/www/wwwroot/mx-work-notes
```

你可以：

- 使用宝塔文件管理器上传代码压缩包再解压
- 或者在终端里直接 `git clone`

推荐命令：

```bash
git clone git@github.com:SuperMua/mx-work-notes.git /www/wwwroot/mx-work-notes
cd /www/wwwroot/mx-work-notes
```

## 6. 环境变量配置

复制：

```bash
cp apps/api/.env.example apps/api/.env
cp apps/web/.env.example apps/web/.env
cp apps/admin/.env.example apps/admin/.env
```

按实际情况修改：

- `apps/api/.env`
- `apps/web/.env`
- `apps/admin/.env`

示例：

API：

```bash
DATABASE_URL=postgresql://dbuser:dbpassword@db-host:5432/smart_notes?schema=public
JWT_SECRET=请替换为强随机密钥
PORT=3000
NODE_ENV=production
```

前台：

```bash
VITE_API_BASE_URL=https://api.example.com
```

后台：

```bash
VITE_API_BASE_URL=https://api.example.com
```

## 7. 安装依赖和构建

在项目目录执行：

```bash
corepack pnpm install
corepack pnpm db:generate
corepack pnpm db:migrate
corepack pnpm build
```

如果你在验收环境需要默认账号，可临时执行：

```bash
SEED_OWNER_EMAIL=admin@example.com SEED_OWNER_PASSWORD=强密码 corepack pnpm db:seed
```

## 8. 前后台站点创建

建议在宝塔中创建两个静态站点：

- `app.example.com`
- `admin.example.com`

然后把构建产物复制过去：

```bash
mkdir -p /www/wwwroot/mx-web
mkdir -p /www/wwwroot/mx-admin
cp -r apps/web/dist/* /www/wwwroot/mx-web/
cp -r apps/admin/dist/* /www/wwwroot/mx-admin/
```

在宝塔站点配置中分别把网站根目录指向：

- `/www/wwwroot/mx-web`
- `/www/wwwroot/mx-admin`

## 9. 单页应用路由配置

前台和后台都是单页应用，宝塔里的 Nginx 配置需要支持回退到 `index.html`。

可在站点配置中为前台和后台加入：

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

否则刷新业务页时容易出现 404。

## 10. API 启动方式

宝塔环境里，推荐用 PM2 或进程守护来启动 API，而不是手动在终端常驻。

启动命令可用：

```bash
cd /www/wwwroot/mx-work-notes
node apps/api/dist/apps/api/src/main.js
```

如果使用 PM2，思路通常是：

```bash
pm2 start apps/api/dist/apps/api/src/main.js --name mx-api
pm2 save
```

## 11. API 反向代理

建议在宝塔中为 `api.example.com` 新建站点或反向代理入口，将请求转发到：

```text
http://127.0.0.1:3000
```

典型 Nginx 配置思路：

```nginx
location / {
    proxy_pass http://127.0.0.1:3000;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

## 12. SSL 证书配置

宝塔方案里，证书通常通过面板界面完成。

建议：

- `app.example.com` 开启 HTTPS
- `admin.example.com` 开启 HTTPS
- `api.example.com` 开启 HTTPS

注意：

- API 一旦切到 HTTPS，前台和后台构建时的 `VITE_API_BASE_URL` 也必须是 HTTPS 地址

## 13. 数据库建议

虽然可以在同机额外折腾数据库，但对于当前项目，还是更建议：

- 使用独立 PostgreSQL 实例
- 将数据库连接信息写入 `apps/api/.env`
- 只允许宝塔服务器访问数据库

## 14. 发布更新流程

推荐流程如下：

1. 在宝塔服务器拉取最新代码。
2. 执行 `corepack pnpm install`。
3. 执行 `corepack pnpm db:generate`。
4. 执行 `corepack pnpm db:migrate`。
5. 执行 `corepack pnpm build`。
6. 覆盖前后台静态目录。
7. 重启 API 进程。
8. 用浏览器做登录与主流程 smoke。

## 15. 常见问题

### 15.1 前台或后台刷新 404

通常是站点没有配置：

```nginx
try_files $uri $uri/ /index.html;
```

### 15.2 API 登录请求失败

通常检查：

- API 进程是否在运行
- 反向代理是否正确转到 `127.0.0.1:3000`
- `VITE_API_BASE_URL` 是否正确

### 15.3 页面能打开但无法登录

通常检查：

- 数据库连接是否正常
- API `.env` 是否正确
- 是否已经执行 migration
- 是否需要执行一次 seed

## 16. 更推荐的适用边界

宝塔方案适合：

- 追求部署效率
- 更偏运维界面化
- 交付后由非纯开发人员维护

如果你追求更标准、更可迁移、更容易自动化的正式环境，仍建议优先看：
